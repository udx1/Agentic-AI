Project objective

Build a realistic local demo of ecommerce support automation

Business goal

  • Demonstrate how a support assistant can answer product and policy questions from trusted context.
  • Keep customer-facing answers concise, grounded, and honest about unsupported requests.
  • Show a path from self-service resolution to human handoff when the bot should not answer alone.

Engineering goal

  • Use a local dataset and local-first persistence so the full workflow is inspectable.
  • Separate catalog, retrieval, agent routing, answer generation, UI, and ticket persistence.
  • Build repeatable validations before introducing more provider-dependent LLM behavior.

Data foundation

Grounded support starts with clean, traceable local data

  • Source samples retained under `data/raw/` for traceability.

Support knowledge base

  • 4 global store policy docs.
  • 145 customer review samples.
  • Seed tickets plus runtime tickets for handoff workflows.
Data05

RAG and answer generation

Retrieve, assess, then answer with citations or ask for more context

Current retrieval

  • 907 chunks persisted in Chroma.
  • Nebius `Qwen/Qwen3-Embedding-8B` embeddings.
  • Hybrid behavior combines vector search and lexical signals.
  • Catalog ranking and availability use structured catalog data directly.

Final answer path

  • LangChain `ChatPromptTemplate` defines the support persona.
  • System prompt sets tone, grounding rules, and local-demo boundaries.
  • User prompt injects the question plus cited source context.
  • Deterministic formatter remains fallback when Claude is unavailable.
RAG06

Agent workflow

LangGraph makes support routing explicit and inspectable

ClassifyPolicy, product, troubleshooting, catalog, escalation, unsupported.
PlanChoose retrieval query, candidate count, and routing path.
RetrieveFetch relevant KB chunks or use structured catalog logic.
AssessCheck whether context is sufficient for a grounded answer.
RespondAnswer, clarify, or prepare handoff metadata.

Important guardrail

Account-specific, order-specific, payment-specific, unsafe product, and support contact requests are handled as handoff candidates instead of hallucinated answers.

Evaluation13

Current status

The Phase 0-9 scope is complete; enhancement track has started

Complete

  • Local storefront and backend APIs.
  • RAG pipeline and Chroma index.
  • LangGraph support agent.
  • Chat UI and eval mode.
  • Ticket persistence and support console.
  • LLM persona prompt template with deterministic fallback.

Known limitations

  • Runtime tickets use local JSON, not a production database.
  • Support console is URL-mode gated, not auth-secured.
  • No real phone, email, SMS, CRM, SLA, or operator assignment integration.
  • OpenAI embeddings remain blocked by quota.
